# Sora ビジョン

## プロダクト名

**Sora**

Sora は「繋がる心が力になる」という思想から名付ける。このプロダクトでは、その思想を技術的かつ創造的なシステムとして具体化する。人間、AI エージェント、DAW、VST インストゥルメント、エフェクト、マニュアル、プロジェクトファイル、MIDI、オーディオ解析、制作知識を、一つの創作ワークフローとして接続する。

Sora は単なる AI 音楽生成ツールではない。Sora は音楽制作のための、接続されたプロダクションレイヤーである。

## コアビジョン

**Sora は、創作意図を接続された制作アクションへ変換する。**

ユーザーは次のように言えるべきである。

> このベースラインに合うメタルギターリフを作って。Heavier7Strings の奏法を使って、MODO DRUM のグルーヴをタイトにして、AmpliTube でギターを整えて、Ozone でラウドだけど潰しすぎないマスターを準備して。

Sora は音楽的意図を理解し、利用可能なプロジェクトコンテキストを確認し、適切なツールを選び、デバイス固有の知識を適用し、MIDI・オーディオ・制御データを生成または変更し、何を変えたのかを説明する。

## プロダクト仮説

現代の音楽制作は、多数のツールに分断されている。

- DAW
- VST インストゥルメント
- VST エフェクト
- MIDI マップ
- キースイッチ
- オートメーションレーン
- プラグインプリセット
- PDF マニュアル
- ステム
- リファレンス
- プロジェクトごとの慣習
- 人間の好み

それぞれのツールには強力な機能がある。しかし、ユーザーが創作意図を低レベルな操作へ手動で翻訳し続けなければならないため、制作フローが分断される。

Sora は、この分断を接続するために存在する。

## キーアイデア

**接続されたコンテキストこそがプロダクトである。**

Sora は、次の要素を接続できるときに価値を発揮する。

- ユーザーが何を作りたいか
- 現在の曲に何が含まれているか
- どのインストゥルメントやプラグインが利用可能か
- 各デバイスをどう制御するか
- どの音楽スタイルを目指しているか
- どの変更が安全か
- それらの変更をどう書き出し、または適用するか

目標はプロデューサーを置き換えることではない。目標は、人間の方向性を保ちながら、技術的な実行を担える、接続された創作パートナーをプロデューサーに与えることである。

## Sora が目指す姿

Sora は、次のことができる AI 制作環境になるべきである。

1. 音楽プロジェクトを理解する。
2. ユーザーの創作意図を理解する。
3. 利用可能なインストゥルメント、エフェクト、マニュアルを理解する。
4. 音楽的に適切な MIDI とオートメーションを生成する。
5. DAW やプラグインに対する操作を実行、または実行可能な形に準備する。
6. レンダリング済みオーディオを解析し、改善案を提示する。
7. 見えない魔法ではなく、レビュー可能な変更を生成する。
8. ユーザーが承認、却下、比較、反復できるようにする。

## Sora ではないもの

Sora は次のようなものではない。

- ブラックボックスな楽曲生成ツール
- ワンクリックのマスタリングプリセット選択機
- 音楽アドバイス用の汎用チャットボット
- DAW の代替
- 特定ベンダー専用の固定プラグインラッパー
- prompt-to-audio のおもちゃ

Sora は、ユーザーの実際の制作環境に根ざしているべきである。

## プロダクト原則

### 1. 自動化より接続

自動化だけでは不十分である。Sora は意図、コンテキスト、ツール、レビュー可能な出力を接続するべきである。すべての操作は、音楽的な理由に追跡できる必要がある。

### 2. 人間の創作コントロール

ユーザーはプロデューサーであり続ける。Sora は提案し、生成し、編集し、説明する。決定するのはユーザーである。

### 3. ハードコードより Device Profile

すべての VST インストゥルメントやエフェクトは Device Profile として表現する。Device Profile は、そのデバイスが何であり、どう制御され、どの音楽的役割を持ち、どんな制約があるかを記述する。

例:

- Heavier7Strings: ギター音源、キースイッチ奏法、演奏可能レンジ、メタルリフの慣習
- MODO BASS: ベース音源、奏法、弦の挙動、演奏スタイル
- MODO DRUM: ドラム音源、MIDI ノートマップ、ベロシティ挙動、キットピース
- AmpliTube: ギター/ベース用エフェクトチェーン、アンプ、キャビネット、ペダル、トーンコントロール
- Ozone: マスタリングエフェクト、モジュール、目標ラウドネス、トーナルバランス、リミッティング挙動

### 4. Project Context を最優先する

Sora は孤立した状態で生成するべきではない。まず Project Context を読み取る、または受け取るべきである。

- BPM
- キー
- 拍子
- セクション構成
- 既存 MIDI
- ステム
- コード情報
- ユーザーメモ
- リファレンストラック
- 現在のプラグイン/デバイス構成

### 5. レビュー可能な成果物

Sora は可能な限り具体的な成果物を出力するべきである。

- MIDI ファイル
- JSON プラン
- オートメーションマップ
- プリセット提案
- パッチ手順
- レンダリング解析レポート
- diff 形式の変更要約

ユーザーは Sora が何をしたのかを確認できるべきである。

### 6. 段階的な制御深度

Sora は、技術的に利用可能な範囲に応じて複数段階の制御をサポートするべきである。制御深度は control level(0〜5)として定義する。

- Level 0: 提案のみ
- Level 1: 手動インポート用ファイルの書き出し(既定)
- Level 2: 仮想 MIDI ポート経由の MIDI/CC 制御
- Level 3: DAW プロジェクトの読み取り
- Level 4: DAW への書き込み(クリップ・オートメーション・トランスポート)
- Level 5: レンダリングと、プロジェクトを理解した完全なエージェント編集

control level はユーザーが設定として宣言する。引き上げはユーザーの明示的な依頼に基づいてのみ行われ、Sora が自発的に制御深度を引き上げてはならない。

アーキテクチャは、初日から最終段階を前提にしてはならない。

## 概念モデル

Sora は 5 つのレイヤーを接続する。

```text
User Intent
  -> Project Context
  -> Music Reasoning
  -> Device Profiles
  -> DAW / Plugin Control
  -> Reviewable Output
```

### User Intent

ユーザーからの自然言語リクエスト。

例:

- このサビをもっと重くして。
- ベースに沿ったシンコペーションのギターリフを足して。
- ドラムをもっと人間っぽくして。
- パームミュートとピッキングハーモニクスを使って。
- スネアを潰さずに配信用にマスターして。

### Project Context

現在の曲に関する構造化された理解。

例:

- テンポ
- キー
- コード進行
- セクション
- 既存 MIDI ノート
- オーディオステム
- 現在のプラグインチェーン
- トラックの役割

### Music Reasoning

Sora が行う音楽的判断。

例:

- どこに余白を残すか
- どこでベースをダブルするか
- どこにリズムの対比を加えるか
- どの奏法がフレーズに合うか
- ギタートーンをどの程度攻撃的にするか
- マスターをどの程度ラウドにするか

### Device Profiles

インストゥルメントやエフェクトに関する再利用可能な知識。

例:

- MIDI ノートマップ
- キースイッチマップ
- コントロールチェンジマップ
- パラメータ名
- プリセットカテゴリ
- 安全な値域
- ジャンル固有のデフォルト

### DAW / Plugin Control

実行レイヤー。

例:

- MIDI クリップを作る
- MIDI ファイルを書き出す
- MIDI ノートを送信する
- CC 値を送信する
- オートメーションを書く
- DAW スクリプトを呼び出す
- プリセットを選択する
- レンダリングを要求する
- オーディオを解析する

### Reviewable Output

Sora が生成するものは、すべて検査可能であるべきである。

例:

- 生成された MIDI
- 編集プラン
- 奏法の説明
- ミックス/マスターレポート
- A/B バリエーション
- 実装メモ

## 戦略的方向性

Sora の最初のバージョンは、すべてを直接制御しようとするべきではない。まずは、創作意図を構造化された制作成果物へ変換することに優れるべきである。

推奨される進行:

### Phase 1: ファイルベースのアシスタント

- 入力: MIDI、オーディオ、プロジェクトメモ、マニュアル
- 出力: 生成 MIDI、奏法マップ、ミックス提案、マスタリングプラン
- ライブ DAW 制御は不要

### Phase 2: Device Profile システム

- 一般的なデバイス向けの構造化プロファイルを追加する
- Heavier7Strings、MODO BASS、MODO DRUM、AmpliTube、Ozone をサポートする
- マニュアルをコンパクトな操作知識へ変換する

### Phase 3: Claude Code + MCP 制御

- Sora が MCP サーバーとして `analyze_project`、`read_midi`、`compose_part`、`apply_articulations`、`export_midi`、`send_midi`、`suggest_plugin_settings` などのツールを公開する
- 仮想 MIDI とファイル書き出しを実行ブリッジとして使う

### Phase 4: DAW-aware ワークフロー

- 可能な範囲でプロジェクト状態を読む
- 可能な範囲でクリップやオートメーションを書く
- 解析用にステムやミックスをレンダリングする
- 反復的な A/B ワークフローをサポートする
- 制御できない操作は、常にファイル書き出しへフォールバックできるようにする

### Phase 5: 制作コパイロット

- Sora は完全な創作制作パートナーになる
- アレンジ、演奏、音作り、ミックス、マスターを横断して判断できる
- それでもユーザーをコントロールの中心に置き続ける

## North Star Experience

ユーザーがこう言う。

> ヴァースのベースラインは良いけど、曲にもっと重いモダンメタルのセクションが必要。タイトなギターリフをパームミュート込みで足して、ドラムをそれに合わせて、ボーカルの余白は残して、2バージョン出して。

Sora は次を生成する。

- ベースライン解析
- 2つのリフ案
- キースイッチ付き Heavier7Strings MIDI
- ベロシティ/ヒューマナイズ付き MODO DRUM MIDI
- AmpliTube チェーン提案
- ミックスメモ
- レビュー可能なファイル
- 音楽的判断の明確な説明

ユーザーは聴き、バージョンを選び、改善を依頼する。

それが Sora である。
