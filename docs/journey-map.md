# Sora ジャーニーマップ

## 目的

このドキュメントは、Sora の想定ユーザージャーニーを、初期セットアップから成熟した日常利用まで記述する。Claude Code での技術計画に使うことを目的とする。

Sora は段階的なシステムとして構築するべきである。完全な DAW 自動化が存在する前から価値を提供しつつ、深いプロジェクト理解と制御へ進化できる道筋を残す。

## 主要ユーザー

初期ユーザーは、次のような個人の音楽クリエイターまたはプロデューサーである。

- DAW 内で作業している
- VST インストゥルメントとエフェクトを使っている
- 詳細なプラグインマニュアルや PDF を持っている
- 作曲、編曲、音作り、ミックス、マスタリングに AI 支援を求めている
- ブラックボックス生成よりも、レビュー可能な技術的出力を好む
- Claude Code を使ったツールオーケストレーションに抵抗がない

## Journey stage 1: 環境オンボーディング

### ユーザーの目標

Sora に自分の制作環境を伝える。

### ユーザーの行動

ユーザーは次を提供する。

- DAW 名とバージョン
- OS
- 利用可能な VST インストゥルメント
- 利用可能な VST エフェクト
- プラグインマニュアルまたはドキュメント
- 好みのジャンル
- 典型的なワークフロー
- プロジェクトフォルダの慣習

例:

> Fender Studio Pro を使っている。Heavier7Strings、MODO BASS、MODO DRUM、AmpliTube、Ozone がある。モダンメタルと J-rock を作る。マニュアルはこのフォルダにある。

### Sora の責務

Sora は次を行うべきである。

- environment profile を作成する
- 利用可能なデバイスを特定する
- 不足しているが重要な制御情報がある場合のみ質問する
- マニュアルを Device Profile 候補へ変換し、機械検証(スキーマ + 整合性チェック)を通す
- デバイス能力を構造化された形式で保存する
- 各 Device Profile の検証用 MIDI(全奏法を 1 音ずつ鳴らす `.mid`)を書き出し、ユーザーが実機確認できるようにする

### 技術成果物

想定される出力:

- `sora.config.json`(control level の宣言を含む)
- `devices/heavier7strings.profile.json`
- `devices/modo-bass.profile.json`
- `devices/modo-drum.profile.json`
- `devices/amplitube.profile.json`
- `devices/ozone.profile.json`
- `devices/*.verify.mid`(奏法マッピングの検証用 MIDI)

### 成功基準

ユーザーが Sora に、どのデバイスが利用可能で、それらをどう制御できるか質問できる。検証用 MIDI により、キースイッチが実機で意図通り発音することを確認できる。

## Journey stage 2: 最初の音楽コンテキスト取り込み

### ユーザーの目標

Sora に実際の音楽的な出発点を与える。

### ユーザーの行動

ユーザーは次のうち 1 つ以上を提供する。

- ベースライン MIDI
- ドラム MIDI
- ギター MIDI
- オーディオステム
- ラフバウンス
- BPM/キーのメモ
- セクションラベル
- リファレンス説明

例:

> これはヴァースのベースライン。これに合うメタルギターリフを作って。

### Sora の責務

Sora は次を行うべきである。

- 可能な範囲で MIDI またはオーディオメタデータを解析する
- 可能な範囲でテンポ、拍子、ノート密度、リズム、調性中心を推定する
- トラックの役割を特定する
- 現在の音楽素材を要約する
- 曖昧さが有用な出力を妨げる場合のみ確認する
- 不足しているが出力品質に効くコンテキストがあれば、提供方法のヒントを添えて知らせる(提供を強制はしない)

### 技術成果物

想定される出力:

- `project-context.json`(セクション構成は `sections` フィールドとして保持する)
- `analysis/bassline-analysis.md`

### 成功基準

Sora が、関連するパートを生成できる程度に音楽素材を正確に説明できる。

## Journey stage 3: 最初の生成パート

### ユーザーの目標

特定のインストゥルメント向けに有用なパートを生成する。

### ユーザーの行動

ユーザーは制作リクエストを行う。

例:

> ベースラインに合う Heavier7Strings のリフを作って。パームミュートを使って、フレーズ終わりにピッキングハーモニクスを入れて。

### Sora の責務

Sora は次を行うべきである。

- スタイルと制約を解釈する
- 対象 Device Profile を選択する
- 音楽的に一貫した MIDI を生成する
- Device Profile を使って奏法を適用する
- レビュー可能なファイルを書き出す
- フレーズ設計を説明する

### 技術成果物

想定される出力:

- `exports/guitar-riff-v1.mid`
- `exports/guitar-riff-v1.plan.json`
- `exports/guitar-riff-v1.md`

### 成功基準

ユーザーが MIDI を DAW にインポートし、正しい奏法を含む音楽的に関連したパートを聴ける。

## Journey stage 4: 複数楽器のアレンジ

### ユーザーの目標

Sora に複数の楽器を連携させる。

### ユーザーの行動

ユーザーは接続されたアレンジ変更を依頼する。

例:

> ドラムをこのリフに合わせて、低域がタイトに保たれるようにベースも調整して。

### Sora の責務

Sora は次を行うべきである。

- 複数トラックを横断して判断する
- ベースとギターのリズム衝突を避ける
- ドラムのサポートパターンを作る
- バリエーションとフィルを追加する
- 求められた場合はボーカルやリード楽器の余白を残す
- 別々にレビュー可能なパートを出力する

### 技術成果物

想定される出力:

- `exports/guitar-riff-v2.mid`
- `exports/bass-adjustment-v1.mid`
- `exports/drums-v1.mid`
- 各 `.mid` に対応する `.plan.json`(Part Plan)
- `arrangement-plan.md`

### 成功基準

生成されたパートが、独立したループではなく、互いに一緒に存在しているように聴こえる。

## Journey stage 5: トーンとプラグインガイダンス

### ユーザーの目標

MIDI 作曲から音作りへ進む。

### ユーザーの行動

ユーザーはトーン変更を依頼する。

例:

- ギターをもっとモダンメタルっぽくして。
- 低域をタイトに保って。
- ベースを歪ませつつコントロールして。
- キックをもっとクリック感のある音にして。

### Sora の責務

Sora は次を行うべきである。

- 適切なエフェクトデバイスを選択する
- プラグイン設定またはプリセットカテゴリを提案する
- なぜその設定がリクエストに合うか説明する
- 明示的に求められない限り、危険または極端な値を避ける
- 後でオートメーションやプラグイン制御に変換できるパラメータプランを作る

### 技術成果物

想定される出力:

- `tone/guitar-amplitube-plan.md`
- `tone/guitar-amplitube-settings.json`
- `tone/bass-chain-plan.md`
- `tone/*.automation.json`(Automation Plan。Phase 4 で `write_automation` の入力になる)

### 成功基準

ユーザーが設定を手動で適用し、意図した方向への変化を聴ける。

## Journey stage 6: レンダリングとオーディオ解析

### ユーザーの目標

Sora が MIDI と意図だけでなく、実際のオーディオに反応できるようにする。

### ユーザーの行動

ユーザーは次を提供する。

- レンダリング済みステム
- ラフミックス
- マスターバウンス

例:

> ラフミックスはこれ。何が悪いか教えて、変更案を出して。

### Sora の責務

Sora は次を行うべきである。

- ツールが利用可能な場合、ラウドネス、スペクトルバランス、ピーク、ダイナミクス、大まかな周波数問題を解析する
- 指定されたジャンルや目標と比較する
- ミックス変更を提案する
- 音楽的な問題と技術的なミックス問題を分ける
- 実行可能な次のステップを出す

### 技術成果物

想定される出力:

- `analysis/rough-mix-report.md`
- `analysis/loudness.json`
- `analysis/mix-fix-plan.md`

### 成功基準

ユーザーが汎用的な助言ではなく、具体的なミックスアクションを受け取れる。

## Journey stage 7: マスタリングワークフロー

### ユーザーの目標

Ozone などのツールを使って、Sora にマスターの準備をさせる。

### ユーザーの行動

ユーザーはターゲットを依頼する。

例:

- 配信用にマスターして。
- ラウドにしたいが、スネアのトランジェントは残して。
- 最終商用マスターではなく、デモ用マスターを作って。

### Sora の責務

Sora は次を行うべきである。

- 目標ラウドネスとダイナミクス目標を決定する
- Ozone のモジュールチェーンを提案する
- 設定と理由を提案する
- 過度なリミッティング、耳に痛い高域、低域の蓄積などのリスクを特定する
- 有用な場合は A/B バリエーションを作る

### 技術成果物

想定される出力:

- `tone/ozone-mastering-plan.md`
- `tone/ozone-settings.json`
- `analysis/mastering-report.md`

### 成功基準

ユーザーが、実際のミックスと用途に結びついた実用的なマスタリングプランを受け取れる。

## Journey stage 8: DAW 制御統合

### ユーザーの目標

手動インポートやパラメータ入力を減らす。

### ユーザーの行動

ユーザーは Sora に制御経路を与える。

- ファイル書き出し/インポート
- 仮想 MIDI ポート
- DAW スクリプティング
- MCP ツール
- プラグインパラメータブリッジ

### Sora の責務

Sora は次を行うべきである。

- 安全なアクションをツールとして公開する
- デフォルトで破壊的編集を避ける
- 制御深度は `sora.config.json` の `control_level`(0〜5)に従い、引き上げはユーザーの明示的な依頼に基づいてのみ行う(Sora が自発的に引き上げない)
- 素材を変更する前にバージョンを作成する(DAW 書き込みは実行前バックアップ + undo 情報の記録を伴う)
- 影響の大きい変更は確認またはプレビューする(書き込み前に仮想 MIDI 試聴を提案する)
- 操作ログを保持する

### 技術成果物

想定されるツール:

- `analyze_project`
- `read_midi`
- `compose_part`
- `apply_articulations`
- `export_midi`
- `send_midi`
- `suggest_plugin_settings`
- `analyze_audio`
- `read_daw_project`
- `daw_transport`
- `write_clip`
- `write_automation`
- `render_stem`

各ツールには要求 control level が定義され、全呼び出しが `logs/actions.jsonl` に記録される。

### 成功基準

ユーザーが自然言語から DAW 上の変更適用まで、最小限の手作業で進められる。

## Journey stage 9: 反復的な制作パートナー

### ユーザーの目標

実際の制作セッション中に Sora を繰り返し使う。

### ユーザーの行動

ユーザーは聴いた後にフィードバックを与える。

例:

- Version B の方が良いけど、後半をもう少し空けて。
- ドラムは良いけど、キックパターンがリフと喧嘩している。
- マスターが明るすぎる。
- サビ前にブレイクダウンを足して。

### Sora の責務

Sora は次を行うべきである。

- プロジェクト内の現在の反復コンテキストを保持する
- 「このリフ」のような指示語を、プロジェクトコンテキスト上のアクティブな素材へ解決し、解決した対象(ファイル)を応答で明示する。曖昧な場合は確認する
- 最初から作り直すのではなく、以前の出力を変更する
- バリエーションを生成する
- トレードオフを説明する
- バージョン管理された成果物を保持する

### 技術成果物

想定される出力:

- `versions/v1/`
- `versions/v2/`
- `CHANGELOG.md`
- `decision-log.md`

### 成功基準

Sora が一回限りの生成器ではなく、制作コラボレーターのように感じられる。

## 理想的な end-to-end ジャーニー

1. ユーザーがツールとマニュアルを登録する。
2. Sora が Device Profiles を作る。
3. ユーザーがベースラインを提供する。
4. Sora が音楽コンテキストを解析する。
5. ユーザーがメタルギターリフを依頼する。
6. Sora が奏法付き Heavier7Strings MIDI を生成する。
7. ユーザーがサポートするドラムとベース修正を依頼する。
8. Sora が連携した MODO DRUM と MODO BASS パートを生成する。
9. ユーザーがトーン方針を依頼する。
10. Sora が AmpliTube とミックス設定を提案する。
11. ユーザーがラフミックスをレンダリングする。
12. Sora がオーディオを解析する。
13. ユーザーがマスタリングを依頼する。
14. Sora が Ozone 設定とバリエーションを提案する。
15. ユーザーはトラックが成立するまで反復する。

## UX トーン

Sora は次のように感じられるべきである。

- 接続されている
- 音楽的に理解している
- 技術的に地に足がついている
- 落ち着いている
- 正確である
- 協働的である
- 非破壊的である

Sora は次のように感じられてはならない。

- ランダム生成器
- プリセットを大量に押し付けるもの
- ブラックボックス
- 過信したミックスエンジニア
- ユーザーと喧嘩する DAW

## MVP ジャーニー推奨

最初に有用なジャーニーは次である。

1. ユーザーが MIDI ベースラインを提供する。
2. Sora が Heavier7Strings の Device Profile を読む。
3. Sora が検証用 MIDI を書き出し、ユーザーが奏法の発音を実機確認する。
4. Sora が互換性のあるギターリフを Part Plan として設計し、MIDI へコンパイルする。
5. Sora が MIDI と理由メモを書き出す。
6. ユーザーが MIDI を DAW にインポートする。
7. ユーザーがフィードバックする。
8. Sora が version 2 を生成する。

このジャーニーは、完全な DAW 自動化を必要とせずに、プロダクトのコアアイデアを検証する。
