# Sora ユースケース

## 目的

このドキュメントは、Sora の具体的なユースケースを定義する。各ユースケースは Claude Code のワークフロー、または将来の MCP ツールフローとして実装可能であるべきである。

ユースケースは、おおよそ初期 MVP から将来的な高度機能の順に並べる。

## 共通概念

### Device Profile

VST インストゥルメントまたはエフェクトに関する構造化された記述。

Device Profile には次が含まれ得る。

- デバイス名
- デバイスタイプ
- 音楽的役割
- MIDI ノートレンジ
- キースイッチマップ
- MIDI CC マップ
- ドラムノートマップ
- 奏法
- プリセットカテゴリ
- パラメータ名
- 安全なパラメータ範囲
- 既知の制約
- マニュアル参照
- オクターブ表記基準(`octave_convention`。C3=60 か C4=60 かを必ず宣言する)
- フィールド単位の確信度(`confidence`: verified / manual / unverified)

### Project Context

現在の曲またはセッションに関する構造化された記述。

Project Context には次が含まれ得る。

- BPM
- キー
- 拍子
- セクションラベル
- MIDI トラック
- オーディオステム
- コード進行
- 現在のプラグインチェーン
- ユーザーの意図
- リファレンスメモ
- 値の出所(`confidence`: stated / estimated / daw)
- アクティブ素材ポインタ(`active_source`: 「このリフ」等の指示語が指す、トラックごとの現在の正)

### Part Plan(中間表現)

Sora は MIDI バイナリを直接生成しない。まず Part Plan(人間可読な JSON、`*.plan.json`)を生成し、決定論的なコンパイル工程が Device Profile を参照して検証・奏法解決・`.mid` 出力を行う。オートメーションも同型の Automation Plan(`*.automation.json`)を経由する。

Part Plan がレビュー・diff・バージョン管理の単位である。

### Control Level

実行系の操作は `sora.config.json` の `control_level`(0〜5)で段階的に有効化される。引き上げはユーザーの明示的な依頼に基づいてのみ行われ(ユーザーが自然言語で依頼し、Sora が変更コマンドを実行する形を含む)、Sora が自発的に引き上げることはない。

### Reviewable Output

Sora は、確認、インポート、diff、バージョン管理が可能な出力を優先するべきである。

例:

- `.mid`
- `.plan.json` / `.automation.json`(Plan 系 JSON)
- `.json`
- `.md`
- `.wav` 解析レポート
- オートメーションデータ
- プラグイン設定プラン

## Use case 1: ベースラインからメタルギターリフを生成する

### ユーザーリクエスト

> これがベースライン。Heavier7Strings 用にモダンメタルのギターリフを作って。パームミュートを使って、フレーズ終わりにピッキングハーモニクスを入れて。

### 入力

- ベースライン MIDI
- BPM
- キー、または推定された調性中心
- Heavier7Strings Device Profile
- スタイル説明

### Sora のアクション

1. ベースライン MIDI を解析する。
2. リズムモチーフと調性中心を特定する。
3. ギターがベースをダブルするべきか、補完するべきか、対比するべきかを判断する。
4. リフの Part Plan(`guitar-riff-v1.plan.json`)を生成する。
5. コンパイル工程が Heavier7Strings の奏法をキースイッチまたはマップ済みコントロールへ解決する。
6. 検証(音域・キースイッチ衝突)を通過した MIDI を書き出す。
7. リフ設計を説明する。

### 出力

- `guitar-riff-v1.mid`
- `guitar-riff-v1.plan.json`
- `guitar-riff-v1.md`

### 受け入れ基準

- MIDI が DAW に正常にインポートできる。
- ノートが妥当なギターレンジ内にある。
- 対象プロファイルに対して奏法が正しく表現されている。
- リフのリズムがベースラインと関連している。

## Use case 2: 2つ目のリフバリエーションを生成する

### ユーザーリクエスト

> Version 1 は忙しすぎる。ボーカルの余白が増えるように、もっとシンプルなバージョンにして。

### 入力

- 前回のリフ出力
- ユーザーフィードバック
- 元のベースライン
- Project Context

### Sora のアクション

1. 前回バージョンを読み込む。
2. 密度とフレーズ境界を特定する。
3. ノート密度を下げる。
4. 中核となるグルーヴを維持する。
5. ボーカルが入りそうな位置に余白を追加する。
6. 新しいバージョンを書き出す。

### 出力

- `guitar-riff-v2.mid`
- `guitar-riff-v2.plan.json`
- `change-summary.md`

### 受け入れ基準

- Version 2 が Version 1 と関連していることが分かる。
- アレンジにより多くの余白がある。
- 変更要約が、何をシンプルにしたか説明している。

## Use case 3: ギターリフから MODO DRUM パートを作成する

### ユーザーリクエスト

> このリフを支える MODO DRUM を作って。ヴァースはハーフタイム感、サビは開ける感じにして。

### 入力

- ギター MIDI
- セクションマップ
- MODO DRUM Device Profile
- スタイル説明

### Sora のアクション

1. リフのアクセントを解析する。
2. パームミュートのリズムを支えるキックパターンを作る。
3. リクエストされたフィールに従ってスネアを配置する。
4. セクションの強度に合うハイハットまたはシンバルを追加する。
5. セクション遷移にフィルを追加する。
6. 安全な範囲内でベロシティとタイミングをヒューマナイズする。
7. ドラム MIDI を書き出す。

### 出力

- `drums-v1.mid`
- `drums-v1.plan.json`

### 受け入れ基準

- ドラムノートマッピングが MODO DRUM プロファイルと一致している。
- キックがリフを支えつつ、過度に混雑していない。
- セクション間の対比が聴いて分かる。

## Use case 4: MODO BASS パートを作成または調整する

### ユーザーリクエスト

> ギターにロックする MODO BASS パートにして。ただしサビ前だけもう少し動かして。

### 入力

- ギター MIDI
- コード/キーコンテキスト
- MODO BASS Device Profile
- セクションマップ

### Sora のアクション

1. ギターのリズムとルート移動を特定する。
2. 主要アクセントにロックするベースラインを生成する。
3. サビ前に経過的な動きを追加する。
4. プロファイルでサポートされる場合、ゴーストノートやスライドを追加する。
5. MIDI を書き出す。

### 出力

- `bass-v1.mid`
- `bass-v1.plan.json`
- `bass-v1.md`

### 受け入れ基準

- ベースが現実的なレンジに収まっている。
- ベースが低域の明瞭さを支えている。
- 追加された動きが、依頼されたセクションで発生している。

## Use case 5: 既存 MIDI に奏法マッピングを適用する

### ユーザーリクエスト

> このプレーンな MIDI を Heavier7Strings の奏法付きで演奏できるようにして。

### 入力

- プレーンなギター MIDI
- Heavier7Strings Device Profile
- スタイルの好み

### Sora のアクション

1. MIDI を Part Plan へ逆変換(デコンパイル)する。
2. パームミュートに適したゾーン、サステインゾーン、スライド、アクセントを検出する。
3. Part Plan に奏法注釈を加える。
4. 必要に応じてベロシティを調整する。
5. 奏法付き MIDI へ再コンパイルする(キースイッチ衝突はコンパイル時に検査される)。

### 出力

- `articulated-guitar.mid`
- `articulated-guitar.plan.json`
- `articulation-report.md`

### 受け入れ基準

- 奏法が音楽的に妥当である。
- キースイッチ配置が演奏ノートと衝突しない。
- レポートが奏法判断を説明している。

## Use case 6: プラグインマニュアルから Device Profile を作成する

### ユーザーリクエスト

> PDF マニュアルはこのフォルダにある。このプラグインの Device Profile を作って。

### 入力

- プラグインマニュアル PDF または抽出済みテキスト
- プラグイン名
- 対象ロール

### Sora のアクション

1. Agent(Claude Code)が PDF マニュアルを直接読解する(巨大マニュアルはページ範囲で分割読解)。
2. MIDI マップ、キースイッチ、CC、パラメータ名、制約、オクターブ表記基準を特定する。
3. コンパクトな Device Profile として起草する。
4. 不確かな情報を `confidence`(verified / manual / unverified)で明示する。
5. 機械検証(スキーマ + 整合性チェック。キースイッチと音域の衝突等)を通し、不整合を修正する。
6. 検証用 MIDI(全奏法を 1 音ずつ鳴らす `.mid`)を書き出し、ユーザーの実機確認結果で `confidence` を `verified` へ昇格する。

### 出力

- `devices/plugin-name.profile.json`
- `devices/plugin-name.verify.mid`

### 受け入れ基準

- Profile に実行可能な制御知識のみが含まれている。
- 機械検証(スキーマ + 整合性チェック)を通過している。
- `octave_convention` が宣言されている。
- 不確かなフィールドが `confidence` で明示的にマークされている。
- 可能な場合、元マニュアルのセクションが参照されている。

## Use case 7: AmpliTube のギタートーンを提案する

### ユーザーリクエスト

> このギタートーンをモダンメタルにして。低域はタイト、ミッドは攻撃的、でもファズっぽく痛くはしないで。

### 入力

- ギターの役割
- 目標ジャンル
- AmpliTube Device Profile
- 任意のレンダリング済みギターステム

### Sora のアクション

1. トーン目標を解釈する。
2. アンプ、キャビネット、ペダルのカテゴリを提案する。
3. ゲイン、EQ、キャビネット、マイク、ゲートの方向性を提案する。
4. 設定を安全な範囲内に保つ。
5. 期待される音響効果を説明する。

### 出力

- `amplitube-guitar-tone.md`
- `amplitube-settings.json`

### 受け入れ基準

- 提案が手動適用できる程度に具体的である。
- 汎用的な助言に留まらない。
- トーンの方向性がユーザーリクエストに合っている。

## Use case 8: ベーストーンチェーンを提案する

### ユーザーリクエスト

> ベースは歪ませたいけど、サブはクリーンに保って。

### 入力

- ベースステムまたは MIDI コンテキスト
- 利用可能なエフェクト
- MODO BASS Device Profile
- AmpliTube またはその他 FX プロファイル

### Sora のアクション

1. スプリットバンドまたはパラレルのベース処理を推奨する。
2. 低域の基音を保つ。
3. コントロールされた中域の歪みを追加する。
4. EQ とコンプレッション戦略を提案する。
5. 手動設定またはオートメーションプランを作成する。

### 出力

- `bass-tone-plan.md`
- `bass-chain-settings.json`

### 受け入れ基準

- 低域が概念上クリーンに保たれる。
- 歪みが聴こえやすい中域に集中している。
- プランが利用可能なツールで実行できる。

## Use case 9: ラフミックスを解析する

### ユーザーリクエスト

> ラフミックスはこれ。何を直すべき？

### 入力

- ラフミックス音声
- 目標ジャンル
- 任意のリファレンスメモ

### Sora のアクション

1. ツールが利用可能な場合、ラウドネスとピークを解析する。
2. ツールが利用可能な場合、大まかなスペクトルバランスを解析する。
3. 音声が提供されている場合、利用可能なモデル能力で概念的に聴く。
4. アレンジ上の問題とミックス上の問題を分けて特定する。
5. 優先度付きの修正案を提案する。

### 出力

- `analysis/loudness.json`(オーディオ解析ツールの測定値)
- `rough-mix-report.md`
- `mix-fix-priority.json`

### 受け入れ基準

- レポートが具体的で優先度付けされている。
- 助言が音楽的アレンジと技術的ミックスを区別している。
- 提案が実行可能である。

## Use case 10: Ozone マスタリングプランを作成する

### ユーザーリクエスト

> これを配信用にマスターして。ラウドにしたいけど、スネアは潰さないで。

### 入力

- ラフミックス音声
- Ozone Device Profile
- 対象プラットフォームまたはラウドネスの好み

### Sora のアクション

1. マスタリング目標を決定する。
2. Ozone のモジュールチェーンを提案する。
3. EQ、ダイナミクス、イメージング、エキサイター、マキシマイザーの選択を推奨する。

### 出力

### 受け入れ基準

- プランがトランジェントの意図を保っている。
- ラウドネス目標が明示されている。
- 設定が後で自動化できる程度に構造化されている。

## Use case 11: A/B アレンジバリエーションを作成する

### ユーザーリクエスト

> 2バージョン出して。1つはもっとタイトに、もう1つはもっとカオスに。

### 入力

- 現在の MIDI パート
- Project Context
- 対象デバイス

### Sora のアクション

1. Variant A と Variant B の目標を定義する。
2. それぞれに合わせてパートを生成または変更する。
3. 曲の中核的なアイデンティティを維持する。
4. バージョン分けされた成果物を個別に書き出す。
5. 違いを説明する。

### 出力

- `variants/tight/`
- `variants/chaotic/`
- `variant-comparison.md`

### 受け入れ基準

- バリエーションが意味のある違いを持っている。
- 両方が Project Context と互換性を持っている。
- ユーザーが選択または組み合わせできる。

## Use case 12: 編集前に制作変更プランを作成する

### ユーザーリクエスト

> 何か変える前に、このサビをもっと強くするなら何をするか教えて。

### 入力

- Project Context
- オーディオまたは MIDI 素材
- 目標スタイル

### Sora のアクション

1. 現在のサビを解析する。
2. 音楽的および音響的な弱点を特定する。
3. 具体的な変更を提案する。
4. 安全な変更と実験的な変更を分ける。
5. ファイル生成や操作適用の前にユーザー承認を待つ。

### 出力

- `chorus-impact-plan.md`

### 受け入れ基準

- プランが明確で非破壊的である。
- ユーザーが個別アクションを承認できる。
- 提案が現在の素材に結びついている。

## Use case 13: MCP 経由で DAW トランスポートを制御する

### ユーザーリクエスト

> サビから再生して、次のテイクを録って。

### 入力

- DAW 制御の利用可否(DAW アダプタの capabilities 申告)
- control_level 4 以上の設定
- プロジェクトロケータまたはセクションマップ

### Sora のアクション

1. 対象位置を解決する。
2. アダプタがサポートしている場合、再生ヘッドを移動する。
3. 再生または録音を開始する。
4. アクションをログに記録する。

### 出力

- 操作結果
- `logs/actions.jsonl` への操作記録

### 受け入れ基準

- 明示的なユーザーリクエストなしに破壊的操作が発生しない。
- トランスポート操作がログに記録される。

## Use case 14: 仮想ポート経由で MIDI を送信する

### ユーザーリクエスト

> このリフを現在のギタートラックで試聴して。

### 入力

- 生成済み MIDI
- 仮想 MIDI ポート設定
- control_level 2 以上の設定
- 対象トラックルーティング

### Sora のアクション

1. MIDI 出力ルートを検証する。
2. 仮想 MIDI 経由でノートと奏法コントロールを送信する。
3. ノートの鳴りっぱなしを避ける。
4. 終了時にすべてのノートを停止する。

### 出力

- 試聴結果ログ

### 受け入れ基準

- MIDI が意図したインストゥルメントを通して再生される。
- ノートの鳴りっぱなしが防止される。
- ユーザーが別途 MIDI 成果物をインポートできる。

## Use case 16: 既存プリセットのトーンを別デバイスで再現する(トーンマッチング)

### ユーザーリクエスト

> Heavier7Strings に同梱されてるエフェクトプリセットの中からメタルっぽいパラメータ設定のものを特定して、そのエフェクトを AmpliTube で再現して。

### 入力

- 参照元デバイスの Device Profile(プリセットカテゴリ・マニュアル記述)
- 再現先デバイスの Device Profile(例: AmpliTube)
- 参照元プリセットで演奏したレンダリング済みステム(推奨。精度を大きく左右する)

### Sora のアクション

1. プリセット名・カテゴリ・マニュアル記述から、目的(メタル向け等)に合う候補プリセットを特定し、選定理由を説明する。
2. プリセットファイルの内部パラメータは直接読めない前提に立つ(独自バイナリ・非公開形式)。
3. ユーザーが候補プリセットで鳴らしたステムを受け取り、スペクトル・ゲイン・ダイナミクス特性を解析する。
4. 再現先デバイスのパラメータで近似チェーンを提案する。
5. 再現側のレンダリングと参照元ステムを A/B 差分解析し、差を縮める調整案を反復提示する。

### 出力

- `tone/preset-match-plan.md`
- `tone/amplitube-match-settings.json`
- `analysis/tone-match-compare.json`

### 受け入れ基準

- 候補プリセットの選定理由が説明されている。
- 再現提案が音の測定に基づいており、プリセット名の類推だけに頼っていない。
- 反復ごとに参照元との差分が数値で確認できる。

## MVP ユースケース

最初のビルドでは次を優先するべきである。

1. マニュアルから Device Profile を作成する。
2. 検証用 MIDI で奏法マッピングを実機確認する。
3. MIDI ベースラインを読む。
4. ベースラインから Heavier7Strings リフを生成する。
5. 奏法マッピングを適用する。
6. MIDI と説明を書き出す。
7. フィードバックからバリエーションを生成する。

これにより Sora のコアループを検証する。

```text
intent -> context -> device knowledge -> generated artifact -> user feedback -> iteration
```

## 将来的な高度ユースケース

後続バージョンでは次を追加できる。

## 安全性と制御ルール

Sora はすべてのユースケースで次のルールに従うべきである。

1. 非破壊的な出力を優先する。
2. 生成成果物をバージョン管理する。
3. 明示的な承認なしにユーザーのプロジェクトファイルを上書きしない。
4. 変更ログを保持する。
5. Device Profiles の不確実性を明示する。
6. 提案と実行を分離する。
7. 音楽的理由を説明する。
8. ユーザーが生成結果を却下または修正できるようにする。
9. 制御深度は `sora.config.json` の `control_level` でユーザーが宣言し、引き上げはユーザーの明示的な依頼に基づいてのみ行う。Sora が自発的に引き上げてはならない。
10. DAW への書き込みは実行前バックアップと undo 情報の記録を伴う。undo 不能な操作は実行しない。
